# Write-site inventory

Verified inventory of every site that stamps `(dataset_id, data_id, pipeline_run_id)` provenance into the relational ledger (`nodes` / `edges` tables, SQLAlchemy models `cognee/modules/graph/models/Node.py` and `Edge.py`) today. Every claim below was confirmed by reading the file.

The only code that physically writes rows into `nodes`/`edges` is `upsert_nodes` (`insert(Node)`, upsert_nodes.py:74) and `upsert_edges` (`insert(Edge)`, upsert_edges.py:81). Everything else in this table is a *caller* of those two functions and is listed because it is the site that supplies the provenance tuple — i.e. the place Part 1/Part 2 must additionally stamp graph-native provenance. The legacy `GraphRelationshipLedger` writer (`record_data_in_legacy_ledger`) is included for completeness but is dead on the live path.

## add_data_points callers that DO carry dataset/data/run context (transitively reach sites 1–3)

These are not separate write sites — they all funnel through `add_data_points` (row 3) — but they are the ctx-supplying entry points that determine *what* provenance lands, so Part 1/Part 2 must ensure ctx flows correctly from each:

- `cognee/tasks/graph/extract_graph_from_code.py:33` — `add_data_points(chunk_graph.nodes, ctx=ctx)`. Live code-cognify path.
- `cognee/tasks/memify/global_context_index/persist.py:77` — `add_data_points(summary_datapoints, ctx=ensure_global_context_storage_context(ctx))`. memify path.
- `cognee/modules/memify/skill_improvement.py:158,217` — `add_data_points([proposal], ctx=_storage_context(...))`. memify/skill path.
- `cognee/modules/tools/skill_runs.py:131` and `cognee/modules/tools/ingest_skills.py:165` — `add_data_points(..., ctx=_make_storage_context(...))`. Agent-tools path.
- `cognee/modules/retrieval/agentic_retriever.py:359` — `add_data_points(runs, ctx=ctx)`. Retrieval-time write path.
- `cognee/modules/migration/loader.py:480,584` — `add_data_points(..., ctx=ctx)` with ctx adapted by `_provenance_ctx` (loader.py:424-442) to substitute the DataItem's deterministic `data_id` into `ctx.data_item.id`. Migration import path. Note: `_provenance_ctx` does NOT carry `pipeline_run_id` (it builds a SimpleNamespace with only user/dataset/data_item/extras), so migration-loaded artifacts get node/edge ledger rows with `pipeline_run_id=None`.

## Coverage notes (ambiguous / partial / gaps)

- **`pipeline_run_id` is not uniformly stamped.** Site 5 (`coding_rule_associations.py:131`) calls `upsert_edges` without `pipeline_run_id` (defaults None). The migration loader's `_provenance_ctx` (loader.py:424) drops `pipeline_run_id` entirely, so migration-imported nodes/edges land with `pipeline_run_id=None`. Graph-native rollback (`source_run_ref`) cannot scope these to a run. Part 1/2 should decide whether the graph-native equivalent (`SOURCE_RUN_REFS_KEY`) is simply omitted for these or whether ctx must be fixed to carry the run id.

- **Unstamped graph writes that bypass the ledger today.** Several live paths write to the graph store via `graph_engine.add_edges` / `add_nodes` but record NO relational provenance, so they are invisible to today's ledger-based delete/rollback AND would need graph-native stamping to be deletable:
  - `coding_rule_associations.py:125` — `add_data_points(rule_list.rules)` is called WITHOUT `ctx`, so the rule NODES get no ledger rows (only the origin edges at :131 are stamped). The nodes are an unstamped graph write.
  - `coding_rule_associations.py:128` — `graph_engine.add_edges(edges_to_save)` writes the edges to the graph before the conditional upsert; if the ctx guard at :130 fails, those graph edges exist with no ledger row.
  - `global_context_index/persist.py:90` — `persist_context_index_edges` calls `unified_engine.graph.add_edges(summarized_in_edges)` directly ("Structural global context index edges are written directly by this task") with NO upsert_edges call — these SUMMARIZED_IN edges have no relational provenance at all.
  These are gaps relative to the stated goal ("every add_nodes/add_edges path that carries dataset/data/run context"): they carry graph context but never record ledger ownership. Flagging so Part 1/2 can decide whether they need graph-native source refs.

- **`add_data_points` requires all three of user+dataset+data_item to stamp (add_data_points.py:94).** If any is missing (e.g. direct task invocation, or callers passing `ctx=None` like eval-framework task getters and `run_pipeline.py:11`), the upserts are skipped entirely and the graph write proceeds unstamped. The same all-or-nothing guard applies at the dlt site (extract_dlt_fk_edges.py:258-264, additionally requiring pipeline_run_id) and the coding-rules site (requires user+data_item).

- **`custom_edges` are stamped with the same provenance as data-point edges** (add_data_points.py:117-126) — they reuse the single session/commit and the same dataset_id/data_id/pipeline_run_id. Part 1/2 must stamp graph-native provenance on custom edges too (they are added to the graph at :152/:155).

- **`record_data_in_legacy_ledger` is dead on the live path** — confirmed no non-test caller. The legacy `GraphRelationshipLedger` is still *read* (`has_nodes_in_legacy_ledger`, `has_edges_in_legacy_ledger`, `mark_ledger_as_deleted`) and migrated (`namespace_entity_type_node_ids.py:641`), but is not a current write site for new artifacts. It records no dataset/data/run tuple regardless, so it is out of scope for new provenance stamping.

- **No other direct `nodes`/`edges` row inserts exist.** Verified: the only `insert(Node)` / `insert(Edge)` / `session.add_all` against these tables are sites 1, 2, and the legacy ledger (6). All other `add_nodes`/`add_edges` references hit the graph backend adapters, not the relational ledger.
